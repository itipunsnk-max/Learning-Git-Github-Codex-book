Intermediate · ทำงานร่วมกับ branch และ remote

คำสั่งใช้ทำอะไรข้อควรจำ
git branch
git branch <name>		แสดง branch หรือสร้าง branch ใหม่เปลี่ยน branch ด้วย git switch หรือ git checkout
git switch <name>
git checkout <name>		เปลี่ยนไปยัง branch ที่มีอยู่checkout เป็นคำสั่งเก่าที่ยังพบในเอกสาร/โปรเจกต์เดิม
git merge <branch>รวม branch เข้ากับ branch ปัจจุบันตรวจ conflict และ test ก่อน commit merge
git stash
git stash pop
git stash apply		พักงานที่ยังไม่ commit แล้วนำกลับมาapply เก็บรายการไว้; pop นำกลับแล้วลบเมื่อสำเร็จ
git reset --soft <commit>ย้อน HEAD แต่คงงานไว้ใน stagingเหมาะกับ local commit ที่ยังไม่แชร์
git reset --hard <commit>ย้อน HEAD และลบงานใน working treeอันตราย: ตรวจ/สำรองก่อนเสมอ
git pullดึงแล้วรวมการเปลี่ยนจาก remoteใช้ git pull --rebase เมื่อ workflow ของทีมกำหนด
git pushส่ง commit จาก local ไป remoteครั้งแรกใช้ git push -u origin <branch>
git rm <file>ลบไฟล์จาก working tree และ stagingตรวจ git diff --cached ก่อน commit
git branch -m <new-name>
git branch -d <name>		เปลี่ยนชื่อหรือลบ branch localลบ branch หลัง merge แล้วเท่านั้น

Advanced · จัดการประวัติ วิเคราะห์ bug และ release

คำสั่งใช้ทำอะไรใช้เมื่อ
git rebase <branch>นำ commit ของ branch ปัจจุบันไปเรียงต่อบนอีก branchจัดประวัติ local ให้เป็นเส้นตรง; ห้าม rebase ประวัติที่ทีมแชร์แล้วโดยไม่ตกลง
git cherry-pick <commit>หยิบ commit เฉพาะจุดมาใช้บน branch ปัจจุบันbackport fix หรือเลือก patch ที่ต้องการ
git revert <commit>สร้าง commit ใหม่เพื่อยกเลิกผลของ commit เดิมปลอดภัยกว่าสำหรับ commit ที่ push/shared แล้ว
git reflogดูการเคลื่อนที่ของ HEAD และ commit ที่เคยอ้างอิงกู้ commit หลัง reset หรือเปลี่ยน branchผิด
git bisect start
git bisect bad
git bisect good <commit>		ค้นหา commit แรกที่ทำให้เกิด bug ด้วย binary searchต้องมี test/เกณฑ์บอกว่า revision ใด good หรือ bad
git blame <file>ดูว่าแต่ละบรรทัดมาจาก commit ใดตามประวัติ/บริบท ไม่ใช่ใช้กล่าวโทษผู้เขียน
git tag <tag_name>
git tag -a v1.0.0 -m "message"		ติดป้าย release ให้ commitสร้าง tag หลังตรวจ main และ test แล้ว
git push origin --tagsส่ง tag ทั้งหมดไปยัง remoteตรวจว่าไม่มี tag ทดลองหรือข้อมูลที่ไม่ควรเผยแพร่
git gcเก็บกวาดและ optimize object ใน local repositoryให้ Git จัดการอัตโนมัติเป็นส่วนใหญ่; ไม่ใช่คำสั่งแก้ไฟล์หาย
กติกาสำคัญ: คำสั่งที่แก้ประวัติหรือลบงาน เช่น reset --hard, rebase, git rm และ git gc ควรตรวจ git status, สำรอง/สร้าง recovery branch และอ่านผลลัพธ์ก่อนยืนยันทุกครั้ง

01 · BRANCH → COMMIT

UI เส้น branch ที่ประกอบกับตัวอย่างจริง

สมมติว่า repository ชื่อ release-notes เริ่มจากหน้า README แล้วเพิ่มฟีเจอร์ค้นหาใน branch แยก

mainc3f8a21merge: search release notes
feature/search9b12d44feat: add release search
5a0c7detest: cover empty query
อ่านภาพนี้อย่างไร: จุดคือ commit ที่เก็บ snapshot ของงาน ส่วนเส้นคือประวัติที่เชื่อมกัน เมื่อพร้อมจึงรวม feature/search กลับเข้า main เป็น c3f8a21
รายการความหมายในตัวอย่างคำสั่งที่ใช้
branchเส้นงานอิสระที่ทดลองฟีเจอร์ค้นหาgit switch -c feature/search
commitจุดบันทึกที่มีข้อความและผู้แก้git commit -m "feat: add release search"
mergeนำประวัติฟีเจอร์กลับไปยัง maingit switch main
git merge feature/search

02 · INSPECT BEFORE PUSH

ตรวจสอบ commit และประโยชน์ของการดูใน .git

โฟลเดอร์ .git คือฐานข้อมูลภายในของ repository เก็บประวัติ commit, branch, HEAD และข้อมูลที่ใช้ตรวจย้อนกลับได้ การตรวจไม่ได้แปลว่าเปิดไฟล์ข้างในไปแก้ แต่ใช้คำสั่ง Git อ่านอย่างปลอดภัย

# อยู่ที่ root ของ release-notes
git status --short
git diff --check
git diff --cached
git log --oneline --decorate --graph -6
git show --stat --oneline 9b12d44
git fsck --no-reflogs --full
ตรวจอะไรคำสั่งประโยชน์
งานค้างในโฟลเดอร์git statusรู้ว่าไฟล์ใด modified, staged หรือ untracked ก่อน commit
เนื้อหาที่จะส่งgit diff --cachedจับไฟล์เกิน, secret และการเปลี่ยนที่ไม่ตั้งใจ
ประวัติและผู้ปกครองgit log --graphเห็นว่า commit อยู่บน branch ใดและ merge มาจากไหน
รายละเอียด commitgit show 9b12d44ตรวจ diff, author, เวลา และไฟล์ที่เปลี่ยนจริง
ความสมบูรณ์ของฐานข้อมูลgit fsck --fullตรวจ object ใน .git ว่าเสียหายหรือมี object ที่หาไม่เจอหรือไม่
เมื่อใช้ Codex: ให้ขอบเขตชัดเจนว่า “ตรวจ git status, git diff และทดสอบก่อนแก้ไฟล์” จากนั้นอ่าน diff หลังแก้ทุกครั้ง Codex ช่วยวิเคราะห์ได้ แต่คนยังเป็นผู้อนุมัติ commit และ push

03 · END-TO-END FLOW

ตัวอย่าง 1 flow ตั้งแต่ยังไม่มี repo จนเสร็จ

01

ก่อนมี repository

สร้างโฟลเดอร์ วาง README.md และตรวจว่าอยู่ถูกที่

mkdir release-notes
cd release-notes
git init -b main
02

บันทึกจุดเริ่มต้น

stage เฉพาะไฟล์ที่ต้องการ แล้วสร้าง commit แรก

git add README.md
git diff --cached
git commit -m "docs: add release notes readme"
03

แยกงานค้นหา

สร้าง branch เพื่อไม่กระทบ main และให้ Codex ช่วยทำงานที่มีขอบเขต

git switch -c feature/search
# Codex: เพิ่ม search.js พร้อม test
git status
git diff
04

ตรวจ แล้ว commit

ตรวจ diff, รัน test และบันทึกเป็น commit เล็กที่อธิบายได้

git diff --check
npm test
git add search.js search.test.js
git commit -m "feat: add release search"
05

ทบทวนประวัติและรวมงาน

ตรวจ commit 9b12d44 แล้ว merge กลับ main

git show --stat 9b12d44
git switch main
git merge --no-ff feature/search -m "merge: search release notes"
git log --oneline --graph -6
06

ส่งขึ้น remote เมื่อพร้อม

เพิ่ม URL GitHub แล้ว push พร้อมตรวจสถานะปลายทาง

git remote add origin <github-url>
git push -u origin main
git status
ผลลัพธ์สุดท้าย: GitHub มี main ที่รวมฟีเจอร์ค้นหา และในเครื่องมีประวัติที่ย้อนดูได้ตั้งแต่ c3f8a21 กลับไปถึง commit แรก โดยทุกขั้นตรวจสอบก่อนส่ง

04 · PUBLISH CHECKLIST

จาก local repository ขึ้น GitHub และเผยแพร่บทเรียน

ก่อน push จริง ให้เพิ่มไฟล์กันของไม่ควรขึ้น Git และตรวจ remote/สิทธิ์ให้ครบ ขั้นตอนนี้ต่อจาก flow ด้านบนโดยตรง

# สร้าง .gitignore ก่อน stage
@'
.env
*.log
node_modules/
dist/
'@ | Set-Content .gitignore

git status
git add .gitignore docs/
git diff --cached
git commit -m "docs: publish complete git field guide"

# เชื่อม GitHub และส่ง main
git remote add origin https://github.com/<username>/<repository>.git
git remote -v
git push -u origin main
REMOTE

ถ้า remote มีอยู่แล้ว

ใช้ git remote -v ตรวจ URL และ git remote set-url origin <url> แก้ปลายทาง อย่าใส่ token ไว้ใน URL หรือไฟล์

AUTH

ถ้า authentication ไม่ผ่าน

ใช้ Git Credential Manager หรือ SSH ตามนโยบายองค์กร ตรวจบัญชีและสิทธิ์ของ repository แทนการลอง password ซ้ำ

RELEASE

ทำจุดเผยแพร่ให้จำง่าย

หลังตรวจ main แล้วใช้ git tag -a v1.0.0 -m "first complete guide" และ git push origin v1.0.0

PAGES

เผยแพร่เว็บด้วย GitHub Pages

ตั้งค่า Pages ให้ source เป็น branch main และ folder /docs แล้วเปิด docs/index.html เป็นหน้าเริ่มต้น

เกณฑ์เสร็จ: clone จาก GitHub ลงเครื่องใหม่ได้, git status สะอาด, เปิดหน้า docs/index.html ได้, ลิงก์บท 01–07 ไม่เสีย และประวัติ main อธิบายการเปลี่ยนแปลงได้

SAFETY RAILS

เช็กลิสต์ก่อนให้ commit ออกจากเครื่อง

  • git status ไม่มีไฟล์ที่ไม่ตั้งใจรวม
  • git diff --cached ไม่มี token, password หรือข้อมูลส่วนตัว
  • git diff --check ผ่าน และ test ที่เกี่ยวข้องผ่าน
  • ข้อความ commit บอก “ทำอะไร” ไม่ใช่แค่ “แก้แล้ว”
  • ถ้างานถูก push และแชร์แล้ว ให้ใช้ git revert แทนการเขียนประวัติทับด้วย reset
