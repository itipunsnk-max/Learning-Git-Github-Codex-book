Git & GitHub Command Reference

คำสั่งที่ใช้งานจริงบ่อยที่สุด ตั้งแต่ setup ไปจนถึง push/pull และการแก้ปัญหาที่พบบ่อย

1. Setup 2. Workflow ประจำวัน 3. Branch 4. Remote & Push 5. Pull & Fetch 6. แก้ไข/ย้อนกลับ 7. Flag ที่ใช้บ่อย 8. Commit message (Codex-friendly) 9. ปัญหาที่พบบ่อย 10. ตัวอย่างชุดคำสั่ง
01

Setup ครั้งแรก

git config --global user.name "ชื่อของคุณ"
git config --global user.email "email@example.com"
git init

สร้าง repository ใหม่ในโฟลเดอร์ปัจจุบัน

git clone <url>

ดึงโปรเจกต์จาก GitHub มาไว้ในเครื่อง

02

Workflow ประจำวัน

git status

เช็คว่าไฟล์ไหนแก้ไป ไฟล์ไหน staged แล้ว

git add .

เพิ่มไฟล์ที่แก้ไขทั้งหมดเข้า staging area

git add <filename>

เพิ่มเฉพาะไฟล์ที่ต้องการ

git commit -m "ข้อความ commit"

บันทึกการเปลี่ยนแปลง

git commit -am "ข้อความ commit"

รวม add + commit ในคำสั่งเดียว (เฉพาะไฟล์ที่ track อยู่แล้ว)

git log --oneline --graph

ดูประวัติ commit แบบย่อ พร้อมกราฟ branch

03

Branch

git branch

ดูรายชื่อ branch ทั้งหมด

git checkout -b feature/ชื่อฟีเจอร์

สร้าง branch ใหม่แล้วสลับไปทันที

git switch feature/ชื่อฟีเจอร์

สลับ branch (คำสั่งใหม่กว่า checkout)

git branch -d feature/ชื่อฟีเจอร์

ลบ branch ที่ merge แล้ว

git merge feature/ชื่อฟีเจอร์

รวม branch เข้ากับ branch ปัจจุบัน

04

Remote & Push (คำสั่งหลักที่คุณถามถึง)

git remote add origin <url>

เชื่อม repo ในเครื่องกับ GitHub repo (ทำครั้งเดียว)

git remote -v

เช็คว่า remote ชี้ไปที่ไหน

git push -u origin main

-u (--set-upstream) ผูก branch ปัจจุบันกับ origin/main ครั้งแรก หลังจากนี้พิมพ์ git push เฉยๆ ได้เลย

git push -v origin main

-v (--verbose) แสดงรายละเอียดขั้นตอนการ push เพิ่มเติม ใช้ตอน debug

git push -uv origin main

รวมสอง flag ใช้ตอน push branch ใหม่ครั้งแรกพร้อมดู log

git push

push ปกติ (หลังผูก upstream แล้ว)

git push origin --delete feature/ชื่อฟีเจอร์

ลบ branch บน GitHub

เข้าใจ -u ง่ายๆ: ใช้ครั้งแรกที่ push branch นั้นขึ้น GitHub เท่านั้น เพื่อบอก Git ว่า branch นี้ในเครื่อง = branch ไหนบน remote พอผูกแล้วครั้งต่อไปพิมพ์ git push สั้นๆ พอ
05

Pull & Fetch

git pull

ดึงและ merge การเปลี่ยนแปลงจาก remote เข้ามาทันที

git pull --rebase

ดึงมาแบบ rebase แทน merge ทำให้ประวัติเรียบขึ้น

git fetch

ดึงข้อมูลล่าสุดมาดูก่อน ยังไม่ merge เข้า branch ปัจจุบัน

06

แก้ไข / ย้อนกลับ

git restore <filename>

ยกเลิกการแก้ไขไฟล์ที่ยังไม่ได้ add

git restore --staged <filename>

เอาไฟล์ออกจาก staging area (unstage)

git commit --amend -m "ข้อความใหม่"

แก้ข้อความ commit ล่าสุด (ก่อน push เท่านั้น)

git reset --soft HEAD~1

ยกเลิก commit ล่าสุด แต่เก็บไฟล์ที่แก้ไว้

git revert <commit-hash>

ย้อนกลับ commit แบบปลอดภัย (สร้าง commit ใหม่ทับ) ใช้ได้แม้ push ไปแล้ว

git stash

เก็บงานที่ทำค้างไว้ชั่วคราว โดยไม่ต้อง commit

git stash pop

เอางานที่ stash ไว้กลับมา

07

Flag ที่ใช้บ่อย

Flagความหมาย
-uผูก branch ท้องถิ่นกับ remote branch (upstream)
-vverbose แสดงรายละเอียดเพิ่ม
-mใส่ commit message
-aadd ไฟล์ที่ track อยู่แล้วทั้งหมดก่อน commit
-bสร้าง branch ใหม่พร้อม checkout
-d / -Dลบ branch (D = force ลบแม้ยังไม่ merge)
--forceบังคับ push ทับประวัติ remote (ใช้ระวังมาก)
--force-with-leaseforce push แบบปลอดภัยกว่า เช็คก่อนว่าไม่มีใคร push ทับ
08

Commit message เมื่อใช้ Codex / AI ช่วยเขียนโค้ด

รูปแบบ Conventional Commits ที่ใช้กันแพร่หลาย และอ่านง่ายเวลาเช็ค log

Prefixใช้เมื่อ
feat:เพิ่มฟีเจอร์ใหม่
fix:แก้บั๊ก
refactor:ปรับโครงสร้างโค้ด ไม่เปลี่ยนพฤติกรรม
docs:แก้เอกสาร/comment
chore:งานจุกจิก เช่น อัปเดต dependency
test:เพิ่ม/แก้ test
เวลา commit โค้ดที่ Codex หรือ AI ช่วยเขียน: ควรรีวิว diff ด้วยตัวเองก่อนทุกครั้ง (git diff --staged) แล้วเขียน commit message ที่อธิบาย "ทำไม" ไม่ใช่แค่ "อะไร" เช่น git commit -m "fix: แก้ date parsing error สำหรับปี พ.ศ. ใน Power Query"
09

ปัญหาที่พบบ่อยตอน push

"git : The term 'git' is not recognized" (PowerShell)

  1. ติดตั้ง Git จาก git-scm.com/downloads
  2. ปิด VS Code ทั้งหมดแล้วเปิดใหม่ (สำคัญ)
  3. เปิด terminal ใหม่ พิมพ์ git --version เช็คอีกครั้ง

"failed to push some refs" / rejected เพราะ remote มีของใหม่กว่า

git pull --rebase origin main

ดึงของใหม่มา rebase ก่อน แล้วค่อย push อีกครั้ง

Push ครั้งแรกแล้วขอ username/password ทุกครั้ง

GitHub เลิกรับ password แล้ว ต้องใช้ Personal Access Token (PAT) แทน หรือใช้ SSH key เพื่อไม่ต้อง login ซ้ำทุกครั้ง

10

ตัวอย่างชุดคำสั่ง (Workflow Examples)

สร้างโปรเจกต์ใหม่แล้ว push ขึ้น GitHub ครั้งแรก first push

ใช้ตอนมีโค้ดในเครื่องแล้ว แต่ยังไม่เคยเชื่อมกับ GitHub เลย (สร้าง repo เปล่าบน GitHub ไว้ก่อนแล้ว)

# 1. เข้าไปที่โฟลเดอร์โปรเจกต์
cd D:\OneDrive\โปรเจกต์ของฉัน

# 2. เริ่มต้น git ในโฟลเดอร์นี้
git init

# 3. เพิ่มไฟล์ทั้งหมดและ commit ครั้งแรก
git add .
git commit -m "feat: initial commit"

# 4. ตั้งชื่อ branch หลักเป็น main (ถ้ายังไม่ใช่)
git branch -M main

# 5. เชื่อมกับ repo บน GitHub
git remote add origin https://github.com/ชื่อผู้ใช้/ชื่อ-repo.git

# 6. push ครั้งแรก พร้อมผูก upstream
git push -uv origin main

ทำงานประจำวัน: แก้ไข → commit → push daily

workflow ที่ใช้บ่อยที่สุด หลังจาก repo ผูก upstream ไว้แล้ว

# 1. เช็คว่าแก้ไฟล์อะไรไปบ้าง
git status

# 2. ดูรายละเอียดที่แก้ก่อน commit
git diff

# 3. add และ commit
git add .
git commit -m "fix: แก้ date parsing error ในไฟล์ Power Query"

# 4. push ขึ้น GitHub
git push

ทำฟีเจอร์ใหม่ด้วย branch แยก แล้ว merge feature branch

แนะนำเมื่อทำงานร่วมกับคนอื่น หรือไม่อยากให้โค้ดที่ยังไม่เสร็จกระทบ main

# 1. สร้างและสลับไป branch ใหม่
git checkout -b feature/dashboard-filter

# 2. แก้โค้ด แล้ว commit ตามปกติ
git add .
git commit -m "feat: เพิ่มตัวกรองวันที่ในแดชบอร์ด"

# 3. push branch นี้ขึ้น GitHub ครั้งแรก
git push -u origin feature/dashboard-filter

# 4. เมื่อพร้อม merge กลับเข้า main
git checkout main
git pull
git merge feature/dashboard-filter
git push

# 5. ลบ branch ที่ใช้เสร็จแล้ว (เครื่อง + GitHub)
git branch -d feature/dashboard-filter
git push origin --delete feature/dashboard-filter

Clone repo ของทีมมาทำงานต่อ collaborate

ใช้ตอนเข้าร่วมโปรเจกต์ที่มีอยู่แล้วบน GitHub

# 1. clone repo มาไว้ในเครื่อง
git clone https://github.com/ทีม/ชื่อ-repo.git
cd ชื่อ-repo

# 2. ดึงของล่าสุดก่อนเริ่มทำงานทุกครั้ง
git pull

# 3. สร้าง branch ของตัวเองก่อนแก้โค้ด
git checkout -b feature/my-task

# 4. ทำงาน แล้ว push ขึ้นไปให้รีวิว (เปิด Pull Request บน GitHub ต่อ)
git add .
git commit -m "feat: งานของฉัน"
git push -u origin feature/my-task

Push แล้วโดน reject เพราะ remote มีของใหม่กว่า fix rejected push

เกิดเวลาคนอื่น push ก่อน หรือแก้จากเครื่องอื่นไว้

# 1. ดึงของใหม่มาก่อน แบบ rebase ให้ประวัติเรียบ
git pull --rebase origin main

# 2. ถ้ามี conflict ให้แก้ไฟล์ที่ชนกัน แล้วรันต่อ
git add .
git rebase --continue

# 3. push อีกครั้ง
git push

commit ผิด อยากแก้ก่อน push undo

พิมพ์ message ผิด หรือลืมใส่ไฟล์ แล้วยังไม่ได้ push ออกไป

# แก้ข้อความ commit ล่าสุด
git commit --amend -m "fix: ข้อความที่ถูกต้อง"

# ลืมใส่ไฟล์ ให้ add แล้วรวมเข้า commit เดิม
git add ไฟล์ที่ลืม.xlsx
git commit --amend --no-edit
